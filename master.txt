THREE TASKS

TASK 1 — EASY (max_steps: 8, task_id: "easy")

Write a realistic Python function that is a simple data processing utility. It must contain exactly 3 real bugs:

Bug 1 — line ~12: Off-by-one error in a loop. The loop runs range(len(items)) but should run range(len(items) - 1) causing an index out of bounds on the last iteration.

Bug 2 — line ~18: Null check missing. The function accesses item.value directly without checking if item is None first, which crashes when None is passed in a list.

Bug 3 — line ~24: Wrong operator. Uses assignment = inside a conditional check instead of == causing all comparisons to always be True.

No red herrings in easy task.

Ground truth: list of 3 GroundTruthBug objects with exact line numbers matching your code.

Expected agent difficulty: easy task should score around 0.65-0.75 with a baseline LLM. Not 1.0 — that would mean the task is too easy.

TASK 2 — MEDIUM (max_steps: 15, task_id: "medium")

Write a realistic Python web API endpoint handler function. It must contain exactly 4 real bugs, all security-related:

Bug 1 — SQL injection: query built with string concatenation instead of parameterized query.
Bug 2 — Hardcoded secret: API key or password hardcoded as a string literal in the function body.
Bug 3 — Missing input validation: user input passed directly to a downstream function without length or type check, enabling potential overflow or XSS.
Bug 4 — IDOR vulnerability: function accesses a resource using user-supplied ID without checking if the current user owns that resource.

No red herrings in medium task.

Ground truth: 4 GroundTruthBug objects with severity "critical" for SQL injection and IDOR, "major" for hardcoded secret and missing validation.

Expected agent difficulty: medium task should score around 0.40-0.55 with baseline LLM.

TASK 3 — HARD (max_steps: 25, task_id: "hard")

Write a realistic multi-concern Python async service function. It must contain exactly 4 real bugs AND 1 red herring:

Bug 1 — N+1 query: a database call inside a for loop that should be batched outside the loop.
Bug 2 — Race condition: shared mutable state accessed from async code without a lock, causing data corruption under concurrency.
Bug 3 — Memory leak: a resource (file handle or connection) opened inside a try block but not closed in all exit paths — the finally block is missing.
Bug 4 — Silent exception swallowing: a bare except clause that catches all exceptions and does nothing, hiding real errors from callers.
Red Herring — line that looks suspicious (unusual variable name or non-standard pattern) but is actually correct and intentional. If the agent flags this line, it receives a false positive penalty.

Ground truth: 4 GroundTruthBug objects plus 1 GroundTruthBug with is_red_herring=True.

Expected agent difficulty: hard task should score around 0.15-0.30 with baseline LLM. A frontier model that understands async Python and avoids the red herring should score 0.60-0.75.

REWARD ENGINE — env/reward_engine.py

This is your competitive edge. Implement non-sparse rewards:

When operation is "add_comment":
  - Check if line_number is within ±5 lines of any ground truth bug line
  - If match found AND not already commented on this bug:
    + 0.15 base reward
    + 0.05 bonus if severity matches ground truth severity exactly
    + 0.05 bonus if category matches ground truth category exactly
    Maximum per-bug reward: 0.25
  - If match found but already commented on same bug (duplicate):
    - 0.05 penalty
  - If is_red_herring match (agent flagged the trap):
    - 0.20 penalty
  - If no match found anywhere (false positive):
    - 0.10 penalty

When operation is "approve":
  - Check if any critical or major bugs remain unfound
  - If yes: -0.50 massive penalty (approving broken code is the worst real-world outcome)
  - If no critical/major bugs remain: +0.10 bonus

When operation is "request_changes":
  - If bugs found > 0: +0.05 (correct behavior)
  - If no bugs found yet: -0.05 (requesting changes without evidence)

When operation is "done":
  - Trigger grader
  - Return final grader score as the reward for this step

When step limit exceeded without "done":
  - -0.20 penalty added to final score

Efficiency bonus (computed at "done"):
  - If steps_used < 60% of max_steps AND final_score > 0.8: +0.10 bonus

GRADER DESIGN — env/graders/

base_grader.py must implement:

def compute_f1(correctly_identified: int, total_comments: int, total_real_bugs: int) -> float
  - precision = correctly_identified / total_comments (0 if total_comments is 0)
  - recall = correctly_identified / total_real_bugs (0 if total_real_bugs is 0)
  - if precision + recall == 0: return 0.0
  - f1 = 2 * precision * recall / (precision + recall)
  - return round(f1, 4)

def compute_weighted_f1(found_bugs: List[GroundTruthBug], all_bugs: List[GroundTruthBug], total_comments: int) -> float
  - critical bugs have weight 3
  - major bugs have weight 2
  - minor bugs have weight 1
  - nit bugs have weight 0.5
  - compute weighted precision and weighted recall
  - return weighted F1 score between 0.0 and 1.0

Each task grader calls base_grader and returns a single float between 0.0 and 1.0.
The grader must return different values for different inputs — never hardcode a return value.
Run grader 3 times with same input — must return identical result every time (deterministic).

Four endpoints:

POST /reset
- Body: {"task_id": "easy"} (default to "easy" if not provided)

- Returns full CodeReviewObservation as JSON
- Must return HTTP 200

- Body: CodeReviewAction as JSON
- Calls env.step(action)

- No body
- Calls env.state()

- Must return HTTP 200

INFERENCE SCRIPT — inference.py

This script runs an LLM against your environment and produces the mandatory stdout format.

Environment variables to read:
- API_BASE_URL (default: "https://router.huggingface.co/v1")
- MODEL_NAME (default: "Qwen/Qwen2.5-72B-Instruct")
- HF_TOKEN (required — used as API key)

For each of the 3 tasks (easy, medium, hard):

1. Print [START] line
2. Call POST /reset with task_id
3. Build system prompt: "You are an expert Python code reviewer. You will receive buggy code. Your job is to identify real bugs by adding comments with exact line numbers. Be precise — false positives are penalized. When done reviewing, call done."
4. Loop up to max_steps:
   a. Build user message including full_file, code_diff, existing_comments, step_number
   b. Call LLM using OpenAI client (multi-turn — include full conversation history)
   c. Parse LLM response to extract action JSON
   d. Call POST /step with action
   e. Print [STEP] line immediately after step returns
   f. If done is true, break
5. Print [END] line with final score

LLM response parsing: the LLM will return JSON like:
{"operation": "add_comment", "line_number": 12, "severity": "critical", "category": "bug", "message": "Off-by-one error"}
If parsing fails, use operation "done" as fallback and log error.

score in [END] line is the value from info["current_score"] after final step.
success is true if score >= 0.5.

Total runtime constraint: each task must complete in under 6 minutes. Set per-task timeout.
